# Technical Roadmap: "Now, Next, Later" for Production Cycle Maturity

**Date:** 2025-11-21
**Status:** Draft
**Context:** Enhancement of `prod-cycle` observability, dependency management (`check_upstream_updates.sh`), and metrics infrastructure (`emit_metrics.py`).

## Executive Summary

This roadmap outlines the technical evolution required to bridge the gap between our current manual dependency checks and a fully autonomous, observable production cycle. The strategy prioritizes immediate stability and observability ("Now"), moves towards integrated automation ("Next"), and aims for predictive, fully autonomous management ("Later").

## 1. Component Analysis & Strategy

### A. `check_upstream_updates.sh` (Dependency Detection)

*   **Current State:** Runs manually or via cron. Generates a human-readable Markdown report. No structured data output. Fragile error handling.
*   **Strategic Goal:** Transform into a reliable *sensor* that emits structured events which the Agentic Flow system can consume programmatically.

### B. `emit_metrics.py` (Telemetry)

*   **Current State:** Basic script for emitting JSONL logs. Lacks robust typing, schema validation, and error recovery.
*   **Strategic Goal:** Evolve into a robust, typed telemetry library (or robust script) that serves as the canonical interface for all system metrics, enforcing schema consistency.

### C. `af` `prod-cycle` Integration (Orchestration)

*   **Current State:** Focuses on internal Build-Measure-Learn loops. Dependency updates are external and unobserved by the cycle.
*   **Strategic Goal:** Keep the core `prod-cycle` fast and focused on *internal* stability. Treat dependency updates as an *external stimulus* that the cycle observes (via metrics) and reacts to (via specific maintenance modes), rather than blocking every iteration on upstream checks.

---

## 2. "Now" Phase: Reliability & Observability (0-2 Weeks)

**Focus:** Stabilize existing tools, ensure data is structured, and establish visibility.

### 2.1. `check_upstream_updates.sh` Improvements
*   **Structured Output:** Modify the script to generate a JSON report (`upstream_status.json`) in addition to the Markdown report. This allows `governance.py` and other tools to parse status without regex.
*   **Reliability Hardening:**
    *   Add checks for required tools (`npm`, `pip`, `git`) before execution.
    *   Implement simple retries (max 3) for network operations (`git fetch`, `npm outdated`).
    *   Add timeout limits to prevent hangs during automated runs.
*   **Execution:** Continue running via cron (Asynchronous), but ensure logs are placed where `af` can see them.

### 2.2. `emit_metrics.py` Refactoring
*   **Type Safety:** Add Python type hints (`mypy` compatible) for all arguments and functions.
*   **Schema Definition:** Define a `MetricSchema` class or dictionary to validate input arguments against expected types before logging.
*   **Atomic Writes:** Ensure file writes are atomic (write to temp, then rename) to prevent log corruption during concurrent access.
*   **Error Handling:** Wrap file operations in `try/except` blocks with specific error reporting (e.g., `PermissionError`, `IOError`) to stderr, ensuring the parent process knows if logging failed.

### 2.3. `prod-cycle` Integration
*   **Observability Hook:** Update `governance.py` to read the *latest* `upstream_status.json` (generated by the cron job) at the start of a cycle.
*   **Metric Logging:** Use `emit_metrics.py` to log a `dependency_risk` metric into the cycle's telemetry.
    *   *Example:* If `upstream_status.json` shows "Critical Security Vulnerabilities", log `risk_score=high`.
*   **Decision Gate:** If risk is Critical, `governance.py` should log a warning or potentially pause "Autocommit" (Safe Degrade pattern), but *should not* automatically attempt to fix it yet.

---

## 3. "Next" Phase: Automation & Integration (2-6 Weeks)

**Focus:** Automate the "Fix" loop and integrate deeply with the cycle.

### 3.1. Dependency Management Automation
*   **New Command:** Create `af maintenance dependencies`.
    *   Reads the structured JSON report.
    *   Executes safe updates (e.g., `npm update` for minor/patch, `pip install --upgrade`).
    *   Runs the system test suite (`af test`) immediately after updates.
    *   If tests pass: Commits the lockfiles (`package-lock.json`, etc.).
    *   If tests fail: Rolls back changes (`git checkout .`).
*   **Tooling:** Integrate `ncu` (npm-check-updates) for better control over `package.json` upgrades.

### 3.2. `emit_metrics.py` Expansion
*   **Destination Routing:** Support sending metrics to different destinations based on type (e.g., local JSONL for debug, structured event bus for analysis).
*   **Context Enrichment:** Automatically inject environment context (Git SHA, Branch, OS load) so callers don't have to pass everything manually.

### 3.3. `prod-cycle` Integration
*   **Maintenance Mode:** Add a flag `af prod-cycle --mode=maintenance`.
    *   In this mode, the cycle explicitly runs `af maintenance dependencies` as its "Action" step.
    *   This allows the "Orchestrator" agent to schedule a maintenance run when risk is high, without slowing down normal development cycles.

---

## 4. "Later" Phase: Autonomy & Prediction (6+ Weeks)

**Focus:** Predictive capabilities and fully autonomous management.

### 4.1. Autonomous Dependency Agent
*   **Agentic Workflow:** A dedicated sub-agent (Goalie) monitors upstream repositories for relevant changes (not just version bumps, but changelogs).
*   **Smart Updates:** The agent creates a shadow branch, attempts the update, runs `prod-cycle` in that shadow branch, and only merges if the cycle metrics remain healthy (Performance, Stability, Correctness).

### 4.2. Predictive Metrics
*   **Trend Analysis:** `emit_metrics.py` data is ingested into an analytical DB (AgentDB).
*   **Prediction:** "Updates to `dependency-X` have caused regressions 40% of the time." -> System automatically requires human approval for this specific dependency.

### 4.3. `prod-cycle` Evolution
*   **Self-Healing:** If a normal `prod-cycle` fails due to a dependency issue, the system can automatically identify the culprit (using the `dependency_risk` metrics history) and attempt a rollback or pin the version.

---

## 5. Architectural Trade-offs (Running Updates inside `prod-cycle`)

| Feature | Running Inside `prod-cycle` (Synchronous) | Running Asynchronously (Cron/External) | **Recommendation** |
| :--- | :--- | :--- | :--- |
| **Cycle Time** | **Slower.** Checks + Installs add significant latency (minutes). | **Fast.** Zero impact on development cycle time. | **Async** |
| **Stability** | **Lower.** Upstream network flakes or bad releases break the build immediately. | **Higher.** Development environment remains stable until explicit update. | **Async** |
| **Freshness** | **Real-time.** Always using latest code. | **Lagging.** Updates apply at next scheduled window. | **Async + On-Demand** |
| **Debugging** | **Harder.** Failures conflate "my code" vs "their code". | **Easier.** Dependency updates are isolated events. | **Async** |

**Conclusion:** Keep checking and updating **asynchronous** to the core `prod-cycle`. Use `prod-cycle` only to *validate* the new state after an update has been applied, or to *trigger* an update in a dedicated maintenance mode.
